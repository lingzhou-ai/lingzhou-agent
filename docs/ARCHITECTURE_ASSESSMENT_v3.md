# 灵舟 (Lingzhou) 核心架构评估与优化建议 (v3)

## 1. 架构全景
- **循环层 (Loop)**: `tick.py` 驱动认知周期，负责任务调度、探针触发、WM 维护与执行分发。
- **判断层 (Judgment)**: `runtime.py` 组装上下文，`context.py` 管理预算，决策下一步动作与模型路由。
- **感知层 (Perception)**: 探针网络监控环境与自身状态，提供预测误差与异常信号。
- **进化层 (Evolution)**: `evolution.py` 基于失败反馈与错误连击触发热重载。
- **记忆层 (Memory)**: SQLite + FTS5 语义记忆，WM 作为短期高优缓存。

## 2. 优势分析
- **热重载能力**: Python 进程内 `importlib.reload` 允许零停机进化，支持运行时自我修复。
- **上下文预算控制**: `context.py` 的优先级截断机制（Skills > Memories > WM）有效防止 Token 溢出。
- **多模型路由**: Reader/Reasoner/Repair 分层调度，平衡响应速度与推理深度。
- **探针监控**: 灵活的传感器网络提供环境感知，支持 interval/manual 触发。

## 3. 风险与瓶颈
- **Tick 阻塞**: 工具调用串行执行，长耗时操作（如网络请求、复杂计算）阻塞整个认知循环。
- **上下文组装开销**: 每次 tick 重新格式化大量状态（WM, Tasks, History），Token 消耗大且延迟高。
- **进化安全性**: 热重载缺乏沙箱验证，语法错误或逻辑缺陷可能导致进程崩溃。
- **记忆检索局限**: 语义记忆主要依赖向量相似度，缺乏结构化关联，复杂任务下召回率受限。

## 4. 可落地的优化方案
### 4.1 引入异步工具执行管线
- **方案**: 改造 `tick.py` 执行分发逻辑，对无依赖工具调用使用 `asyncio.gather` 并发执行。
- **收益**: 降低单 tick 延迟，提升多探针/多文件读取场景的吞吐量。

### 4.2 增量上下文更新
- **方案**: 在 `runtime.py` 中实现上下文缓存机制，仅重算变更部分（如 WM 新增项、情绪变化），静态部分（Axioms, Soul）复用。
- **收益**: 减少 Judgment 层组装时间，显著降低 Token 消耗。

### 4.3 进化安全护栏
- **方案**: `evolution.py` 增加 `dry-run` 验证：热重载前执行 `py_compile` 语法检查与基础 import 测试。失败则自动回滚。
- **收益**: 防止自进化引入致命错误，提升系统稳定性。

### 4.4 记忆图谱化扩展
- **方案**: 语义记忆节点增加 `relations` 字段，支持显式实体关联（如 `task_id`, `file_path`），优化多跳查询。
- **收益**: 提升跨会话、跨任务的上下文关联能力。
